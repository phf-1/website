#+begin_abstract
This article offers a minimal, practical definition of actors aimed at specification
and implementation. It clearly separates protocol from implementation. A stopwatch
actor serves as a concrete, illustrative example.
#+end_abstract

* Introduction

Building correct software requires a clear specification of /what/ the system must do
and an implementation that shows /how/ to do it [cite:@lamport2002specifying]. The Actor
Model provides a powerful abstraction for concurrent and distributed systems by
treating computation as asynchronous message-passing between independent entities.

* Objective

- Provide a clean, minimal definition of an actor
- Illustrate it with a stopwatch example
- Describe a practical method for defining new actor types

* Actor system

#+ID: 889e352a-3ea6-4be5-bcf2-f6943ee37d16
#+begin_definition
An *actor system* is an abstraction over a network of machines. It models computation
as asynchronous message-passing between actors.
#+end_definition

* Actor

#+ID: aae19666-8753-4527-8f7f-70d68533c58c
#+begin_definition
An *actor* is a concurrent computational entity with:
- a unique, unforgeable *address*,
- private *state*,
- a *script* (behavior) that defines how it reacts to messages.

#+begin_example
Actor :≡ Address × State × Script
#+end_example
#+end_definition

* Address

#+ID: e74b3833-8f59-426a-a26f-1ad0b80d5650
#+begin_definition
An *address* is an opaque, unforgeable identifier that names an actor. Addresses can
only be obtained through creation or by receiving them in messages.
#+end_definition

* State

#+ID: 5d74d0a1-2d29-4254-95d1-f961c1951ac2
#+begin_definition
*State* is the private mutable memory of the actor, typically a value of some concrete
type (record, tuple, map, etc.).
#+end_definition

* Script

#+ID: c796968d-171e-4393-a0af-2d22b0657f80
#+begin_definition
A *script* is the behavior of an actor. It defines how the actor reacts to each
incoming message using the following core operations:

#+begin_example
- send   : Address Any → ∅
- become : Script State → ∅
- self() : Address (the executing actor own address)
- spawn  : Script State → Address
- ■        (halt the actor)
#+end_example
#+end_definition

* send

*send* is the fundamental asynchronous communication primitive in the Actor Model.

#+ID: 3f7c9a2b-1d4e-4f8a-9b2c-7e6d5a1f3b9e
#+begin_definition
*send*

instructs the system to deliver the payload to the actor at the given address. The
operation is non-blocking: the sender continues execution immediately. There are no
guarantees of delivery or ordering.

#+begin_example
send(addr payload)
#+end_example
#+end_definition

* become

#+ID: abfd4a0c-98c9-4248-9015-a50a48e8bc6e
#+begin_definition
*become(script state)* replaces the current behavior and state. The new script and
state will be used to process the next message.
#+end_definition

* spawn

#+ID: 8a2b4c6d-9e1f-4b3a-8d7c-2e5f9a1b6c4d
#+begin_definition
*spawn*

creates a new actor and returns its address.

#+begin_example
spawn : Script → State → Address
spawn(script, initialState)
#+end_example

The newly created actor starts running the given ~script~ with ~initialState~
immediately. The returned address is the only way to communicate with the new actor.
#+end_definition

* receive

#+ID: 971a5148-6d66-4ef6-a980-f9486b62090f
#+begin_definition
*receive* blocks until a message arrives, then pattern-matches it against clauses and
executes the corresponding expression.

#+begin_example
receive
  pat₁ ↦ expr₁
  pat₂ when guard ↦ expr₂
  …
#+end_example
#+end_definition

* Example

We assume the existence of an actor =osc= that satisfies the following protocol:
- =subscribe(addr)=: causes =addr= to receive =<osc, "tick">= every second
- =unsubscribe(addr)=: stops sending ticks to =addr=

A stopwatch is an actor that satisfies this protocol:

- Starts counting =seconds : ℕ= from 0 upon creation.
- Replies to =<sender, "read">= with =<self, seconds>=.
- Halts (■) if it receives any other message.

The implementation is defined as follows:

#+begin_example
script(osc) :≡
  osc#subscribe self()
  loop :≡
    seconds : Nat ↦
      receive
        <sender, "read"> ↦
          send(sender <self(), seconds>)
          become(loop seconds)

        <osc, "tick"> ↦
          become(loop (seconds + 1))

        _ ↦ ■
  loop(0)
#+end_example

We create the actor with:

#+begin_example
sw :≡ spawn(script(osc) 0)
#+end_example

=sw= is a stopwatch — /i.e./ it is an actor that satisfies the stopwatch protocol.

**Proof:**

By induction on the number of messages processed by ~sw~.

- **Base case** (0 messages): Actor starts in state ~loop(0)~ after subscribing. Count = 0 and alive. Holds.
- **Inductive step**: Assume the actor is in state ~loop(n)~ and alive. On next message:
  - ~<sender, "read">~: replies ~<self, n>~ and stays in ~loop(n)~.
  - ~<osc, "tick">~: becomes ~loop(n+1)~.
  - Other: executes ■ (halts).

The protocol is preserved in all cases. By induction, ~sw~ satisfies the stopwatch
protocol.

* Actor construction

#+ID: e813ef21-5758-493d-8d65-0f7188db94d0
#+begin_definition
*Actor construction*

1. *What*: Define the protocol the actor must satisfy.
2. *How*: Define the initial state and script.
3. *Proof*: Prove or at least test that the actor satisfies the protocol.
#+end_definition

* Stopwatch interface

The following helpers provide a convenient way to create and use stopwatches:

#+begin_example
Stopwatch#mk : Oscillator → Stopwatch :≡
  osc ↦ spawn(script(osc) 0)

Stopwatch#read : Stopwatch → ℕ :≡
  sw ↦
    send(sw <self, "read">)
    receive
      <sw, seconds> ↦ seconds
#+end_example

Client example:

#+begin_example
sw1 :≡ Stopwatch#mk(osc)
sw2 :≡ Stopwatch#mk(osc)
t1  :≡ Stopwatch#read(sw1)
t2  :≡ Stopwatch#read(sw2)
print("t1: #{t1} seconds")
print("t2: #{t2} seconds")
#+end_example

* Parallel execution

A smart compiler or runtime may execute independent parts in parallel:

#+begin_example
∥
  1. sw1 :≡ Stopwatch#mk(osc)
     t1  :≡ Stopwatch#read(sw1)
     print("t1: #{t1}")

  2. sw2 :≡ Stopwatch#mk(osc)
     t2  :≡ Stopwatch#read(sw2)
     print("t2: #{t2}")
#+end_example

* Generic protocols

The =Stopwatch#read= function works with *any* actor that satisfies the stopwatch
protocol. This highlights that interfaces in actor systems are defined by protocols
rather than concrete types. This enables greater flexibility and composition.

* Protocol negotiation

Actors can learn and negotiate protocols dynamically. This property allows actor
systems to evolve into complex, adaptive “living systems”
[cite:@hintjens2016livingsystems;@kay2011scaling;@kay2016hnambassadors;@kay2003oop].

* Introducing new actor types

New actor types can be introduced using the following template:

#+begin_src org
,* <Name>
:PROPERTIES:
:TYPE: 0672475a-1dfa-4aa0-93d5-add1c1c995ad
:ID: <UUID>
:END:

<Informal description.>

,** Protocol

<Clear description of accepted messages and responses.>

,** mk : 🞎 ⇢ Name

<Constructor description.>

,** <msg> : Name Args ⇢ Reply

<Message interface description.>
#+end_src

We use dashed arrows =⇢= to indicate that replies are not guaranteed, the recipient may
crash or be struck by lightning, instead of plain arrows =→= that would have had
indicated functions.

* Result

We have provided:
- a minimal yet usable definition of actors
- a clear separation between protocol and implementation
- a concrete stopwatch example with proof
- a practical three-step workflow for constructing new actor types

* Discussion

This definition of the Actor Model achieves its primary goals: it is minimal,
pedagogical, and maintains a clear separation between protocol (specification) and
implementation. The inductive proof technique shown with the stopwatch example
provides a practical way to reason about correctness.

However, the stopwatch actor halts immediately upon receiving any unexpected
message. While this choice simplifies the model and the proof, it is overly rigid for
practical applications. Real-world actor systems typically employ supervision,
logging, or graceful degradation rather than unconditional termination on unknown
messages.

Additional limitations include:
- Absence of timeouts on ~receive~, which can lead to deadlocks in more complex
  scenarios.

- Lack of structured error handling and supervision hierarchies.

- The proof technique presented only addresses sequential message processing and does
  not yet cover fairness, message reordering, or network partitions.

- The core primitives remain low-level; many applications would benefit from
  higher-level patterns built on top of them.

These observations open clear paths for future refinements of the model.

* Conclusion

We have presented a clean and minimal definition of actors based on five core
primitives: ~send~, ~receive~, ~become~, ~spawn~, and ~■~. By putting protocols first and
supporting them with concrete implementations and proofs, this article offers both a
practical methodology and a foundation for building understandable concurrent
systems.

The stopwatch example demonstrates that even with very few primitives, one can
construct useful, provably correct behaviors. This protocol-centric approach
naturally leads to flexible, composable, and evolvable designs.

This work is a first approximation in the spirit of progressive refinement. Future
versions may incorporate session types, supervision primitives, timeouts, and tighter
integration with type theory. Ultimately, we hope this framework helps make
concurrent and distributed programming more accessible, more rigorous, and more
reliable.
